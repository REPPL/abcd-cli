# Lint Contract

Canonical reference for the intent, prompt, and terminology checkers in `internal/core/lint` — the three lint surfaces abcd ships. Defines the lint-code namespace, severity model, and CI integration. **Prerequisite for any intent that extends linting** (currently itd-1, itd-4, itd-25, itd-27, itd-28, itd-31, itd-34, itd-36, itd-37, plus the 2026-05-08 surface-discipline sweep).

This section was added post-audit (2026-05-07) because lint extensions had been declared in 6+ intents without a canonical contract — the audit flagged this as a structural pattern of decisions that hadn't been made. itd-48 (which superseded itd-31; cross-document fidelity auditor, post-audit) extends this contract further with meta-audit checks (R1–R7) but does not change the namespace or severity model. Note: fn-29 shipped the **judgement half only** — the `XD` mechanical lint codes named below are still **reserved/deferred** (see `.work/issues.md` under `[fn-29 follow-up]`).

## 1. Lint-code namespace

Lint codes are 2-letter category prefix + 3-digit numeric suffix: `AAnnn`. Categories:

| Prefix | Category | Tool | Purpose |
|---|---|---|---|
| `IL` | Intent Lint (general) | `internal/core/lint` | Schema, frontmatter, acceptance-criteria shape, cross-references |
| `GL` | Glossary Lint | `internal/core/lint` (consumes glossary) | Term defined / forbidden / cross-context / draft-leakage |
| `GR` | Grill | `internal/core/lint` | Intent has acceptance but never grilled / grill-state inconsistencies |
| `RV` | Review (spec-tied) | future `internal/core/lint` (post-fn-2 task 3) | Review front-matter / verdict enum / SHA validation / size cap |
| `IS` | Issue Ledger | `internal/core/lint` (capture extension) | Issue frontmatter / supersedes / state machine |
| `PQ` | Prompt Quality | `internal/core/lint` | Mechanical agent-prompt checks: `prompt_version` semver, `agents/CHANGELOG.md` heading, `capability_scope.task_classes` enum membership, inline-list syntax, canary-fixture presence (`PQ001`–`PQ006`). Broader structural lint — research footer, example-I/O, canary *execution* — is deferred (see §1 reserved codes). |
| `TM` | Terminology | `internal/core/lint` (new in fn-3 task 1) | Glossary file schema validation |
| `XD` | Cross-Document (post-audit, itd-48 / superseded itd-31; mechanical-half codes **deferred** per fn-29 — see `.work/issues.md` `[fn-29 follow-up]`) | `internal/core/lint --cross-doc` | R1–R7 meta-audit (terminology drift, premise drift, scope leakage, reference rot, naming reservation conflicts, schema/state contradictions, acknowledgement gaps) |
| `MQ` | Memory Quotation (added 2026-05-08, itd-36) | `internal/core/lint` (memory extension) | Per-page bounded quotation budget + cumulative source-coverage ceiling; prevents copyright-laundering via N-page reconstruction |
| `MS` | Memory Source-class (added 2026-05-08, itd-36) | `internal/core/lint` (memory extension) | Source-class single-class advisory + cross-class without weighting note. Closed-enum validation on `source.class` is **DEFERRED** — no reserved code emits it and fn-39 ships none (an invalid class string passes through the class-set derivation untyped) |
| `ML` | Memory Licence (added 2026-05-08, itd-36) | `internal/core/lint` (memory extension) | `external_*` page missing `licence` field; `unknown` is acceptable but must be explicit. SPDX-ID validation is **DEFERRED** — fn-39 ships the presence/`unknown` check only; no code validates the licence VALUE |
| `MG` | Modification Grammar (added 2026-05-08, itd-37) | `internal/core/lint` (spec extension) + `intent-fidelity-reviewer` Role 1 (semantic) | Concreteness lint on `## Modification Grammar` section + boilerplate-detection ("could this describe a different spec?" test) |
| `VR` | Vocabulary Registration (added 2026-05-08, itd-37 Ripple) | `internal/core/lint` | Term introduced in spec's `### Ripple > Vocabulary delta` MUST appear in `02-constraints/04-naming.md` glossary in the same spec |
| `SD` | Surface Discipline (added 2026-05-08, sweep) | `internal/core/lint` (surface-design extension) | Forbidden sub-verb naming (e.g., `<verb> show`, `<verb> stats`, `<verb> list` plain, `<verb> view`) — collapses to bare per the bare-command-as-render discipline |
| `PA` | Phase Anchor (added fn-66, adr-9) | `internal/core/lint` (`SpecLinter`) | Verify-exists on a spec's optional `phase:` frontmatter anchor — the anchor must name an existing phase doc under `.abcd/development/roadmap/phases/`. A spec with no `phase:` is legal (unscheduled, per fn-49.2/adr-9) |
| `PR` | Promote backlink (added fn-30, itd-46) | `internal/core/lint` (`--corpus` + single-file) | Promote-bridge four-field back-link contract between a promoted intent (`source_issue` scalar) and its source issue (`promoted_to` scalar) plus the reciprocal `related_issues`/`related_intents` lists — see the promote bridge in [`04-surfaces/05-intent.md`](../04-surfaces/05-intent.md) and the capture promote sub-verb |
| `FS` | Flow-State Drift (added fn-41, itd-49) | `internal/core/flowstate` (standalone emitter at pre-commit; contract row aggregated into `internal/core/lint --codes`) | Checkpoint `tasks[].runtime.status` vs live flow-state store divergence — status REGRESSION only (a `done` checkpoint over a missing/non-`done` store entry, not recency-exempt). Read-only detector, local pre-commit only (both inputs are git-untracked — no CI gate) |
| `RC` | Review Completeness / spec-side linkage hygiene (added fn-43, itd-53; extended fn-48, itd-?) | `native review-queue gate` (`internal/core/spec`; standalone emitter with its own `--codes` — NOT folded into `internal/core/lint`'s aggregate) | Two sub-families. **Review-completeness, intent-tied** (RC001–RC003): shipped intent lacks a completed fidelity-review block / latest rollup `NOT_MET` / `INCONCLUSIVE` — is there a verdict of record per `intents/shipped/` file. **Spec-side linkage hygiene, DONE-spec-tied** (RC006–RC007): a DONE spec references an itd-N whose intent file is not under `shipped/` (RC006) or does not exist anywhere (RC007) — the delivery-state desync guard. Distinct from `RV`, which is **spec-tied** review-ARTIFACT validation (review front-matter / verdict enum / SHA on the native spec store's review files) |

**Numbering**: `nnn` is allocated within each category (`GL001`, `GL002`, …). Codes are stable once shipped — additions only, never reuse a freed number. The first intent to introduce a code in a category bumps the counter; later intents extending the same category must read this section first to avoid collisions.

**Pending reservations (open specs — full registry rows land with those specs' docs tasks):** none currently. (`RC001`–`RC003`, `MQ003`, and `FS001` — formerly pending here — landed with fn-43 .1, fn-39, and fn-41 respectively; see their delivered rows below.) The `.work/issues.md` note questioning whether the `RC` family was ever formally reserved is reconciled here: the family **was** reserved (the `RC001`–`RC005` rows below — `RC001`–`RC003` delivered by fn-43 .1, `RC004`–`RC005` reserved by fn-43/itd-53). fn-48 .2 allocates the next two free codes (`RC006` warn, `RC007` info) for the spec-side linkage guard, preserving the `RC004`–`RC005` reservations.

**Code status** (delivered vs. reserved):
- `IL001` intent frontmatter fails `intent.schema.json` — generic schema-validation failure on an intent file's frontmatter. Severity: blocker. **Delivered** in fn-8 task .1 (`IntentLinter._check_schema`). A `status:` key specifically is mapped to `IL013` instead (no double-report). Line-precise on the offending frontmatter key (fn-82.3): a present-key error (bad enum / pattern / type on a declared field, or an `additional property` whose key is named) anchors on that key's file line via `_yaml.frontmatter_key_lines`, so it survives `--since-staged`; a `missing required property` error names an ABSENT key and anchors at line 1 (frontmatter start), never a phantom line.
- `IL002` intent missing or malformed `## Acceptance Criteria` — an intent file must carry a `## Acceptance Criteria` section with well-formed Given/When/Then bullets. Severity: blocker. **Delivered** in fn-8 task .1 (itd-1; `IntentLinter`). Intent-file-scoped — native-spec-store spec files use `## Acceptance` (owned by the native spec store / oracle review) and are NOT subject to this check.
- `IL003` intent `kind` inconsistent with its lifecycle directory — the directory↔kind matrix (per intent kind: `drafts/` admits `null`/`standalone`/`bundle-member`; `planned/`/`shipped/` require non-null `standalone`/`bundle-member`; `disciplines/` requires `discipline`). A live file with `kind: discipline` in `drafts/`/`planned/`/`shipped/`, or a `planned/` intent with `kind: null`, fails. The matrix's `spec_id` column is also enforced: a `planned/`/`shipped/` intent must carry a non-null `spec_id` of the `fn-N` shape, and a `drafts/` intent must carry `spec_id: null` (both skipped under `--promote-check` — a draft being promoted is assigned its `spec_id` by the native plan handoff, not before). `superseded/` is exempt — it retains the original kind (its supersession fields are `IL006`). Severity: blocker. **Delivered** in fn-9 task .1 (itd-34; `IntentLinter`). Under `--promote-check`, the effective directory is the prospective `--target-location` (`planned` default, or `disciplines`). See the directory↔kind matrix in `.flow/specs/fn-9-three-intent-kinds-kinddirectory-lint.md`. Line-precise on the `kind` / `spec_id` frontmatter key (fn-82.3): the finding anchors on the offending key's file line (or line 1 when the key is genuinely absent — the violation is then the absence), so it survives `--since-staged`.
- `IL004` broken active bundle group — a `kind: bundle-member` in `planned/`/`shipped/` whose `bundle` is null, whose group (all `planned/`+`shipped/` intents sharing that `bundle` id) has fewer than 2 members, or whose members disagree on `kind`/`spec_id`. The check reads intent frontmatter across the `planned/` and `shipped/` corpus (`drafts/` and `superseded/` excluded — a draft bundle is only proposed, a superseded member carries `bundle_at_supersession` not active `bundle`); a malformed peer file is reported as its own `IL004` finding and the group treated as "could not verify". Severity: blocker. **Delivered** in fn-9 task .1 (itd-34; `IntentLinter`).
- `IL005` discipline-kind constraint violation — a live file with `kind: discipline` outside `disciplines/`/`superseded/` (including `drafts/`), or a `disciplines/` discipline with non-null `spec_id`/`bundle`. A discipline candidate is never written to a live file; it exists only as promote-check staging state under `--target-location disciplines`. ("discipline has no `status:` field" is `IL013` — not duplicated.) Severity: blocker. **Delivered** in fn-9 task .1 (itd-34; `IntentLinter`).
- `IL006` `superseded/` intent missing supersession fields — a `superseded/` intent must carry `superseded_by` and `kind_at_supersession`; a former bundle-member (`kind_at_supersession: bundle-member`) must additionally carry `bundle_at_supersession` and have no active `bundle`. `kind` retains its original value (`superseded` is not a schema kind): it must be non-null and, when `kind_at_supersession` is present, equal to it. Severity: blocker. **Delivered** in fn-9 task .1 (itd-34; `IntentLinter`).
- `IL007` malformed `## Press Release` heading — a heading whose text starts with "Press Release" right after the `#` marker but does not match the canonical `## Press Release` form (wrong level, double space, suffix, hyphen/joined, missing marker space, closing-hash without separator) silently disables the GL002 press-release-voice scan for the whole intent, so it is reported as a structural error. Fence-aware (a near-miss inside a code fence is not flagged); scans the body only. Severity: blocker. **Delivered** in fn-18 task .1 (`IntentLinter`).
- `IL008` missing `## Press Release` section — an intent whose EFFECTIVE kind is not `discipline` (including `kind: null` drafts) must carry a canonical `## Press Release` section; without one, GL002 (which scopes to that section) never runs and the intent dodges the press-release-voice discipline. Fence-aware (a `## Press Release` inside a code fence does not satisfy). `kind: discipline` — and a discipline promotion under `--promote-check --target-location disciplines` — is exempt. Corpus scope: `drafts/`/`planned/`/`disciplines/` (scanned, exempt by kind)/`shipped/`; only `superseded/` is excluded. The finding is LINE-PRECISE, anchored at the frontmatter end / first body line (never line 0), so it survives `--since-staged`. Severity: warn. **Delivered** in fn-82 task .1 (`IntentLinter`).
- `IL009`–`IL010` — reserved for intent_lint general (slug match, frontmatter shape, etc.; note: lifecycle state is encoded by directory location only — no `status:` validation under this range, see `IL013` instead).
- `IL011` cross-phase bundle attempt — `/abcd:intent plan <itd-A> <itd-B> ...` (multi-arg, kind=bundle-member) hard-blocks when the proposed members belong to different phases. A bundle ships as one spec, and one spec belongs to one phase; members scoped to different phases cannot share a spec. This is a plan-time check (the planner knows the target phase), not a frontmatter-field check — phase membership lives editorially in the phase docs' `## Scope`, not in intent frontmatter. Severity: blocker (fixed — see §2). **Delivered** in fn-8 task .6 (`BundleLinter`). See bundle invariant in [`04-surfaces/05-intent.md § 1`](../04-surfaces/05-intent.md#1-intent-ids-kinds-and-lifecycle); reserved by itd-34. Worked example: `intent-capture-discipline` retirement (itd-27 and itd-30 scoped to different phases), 2026-05-07.
- `IL012` malformed `surface_history` entry — every entry in `surface_history` must include `date` (ISO YYYY-MM-DD), `from` (free-form surface shape descriptor — e.g., `top-level-skill`, `top-level-command`, `subverb-of-X`, `flag`), `to` (same shape), and `reason` (non-empty). Severity: warn (it's an audit trail, not a gate). **Delivered** in fn-8 task .1. See template in [`04-surfaces/05-intent.md § 3`](../04-surfaces/05-intent.md#3-press-release-format-standalone--bundle-member-kinds) and the two-history-fields explanation in [§ 1](../04-surfaces/05-intent.md#1-intent-ids-kinds-and-lifecycle). Worked example: itd-27's skill → sub-verb demotion on 2026-05-07.
- `IL013` `status:` field present in intent frontmatter — any intent file containing a `status:` key fails lint. Severity: blocker. **Delivered** in fn-8 task .1. Per the 2026-05-08 directive: lifecycle state is encoded by directory location only (`drafts/` / `planned/` / `shipped/` / `disciplines/` / `superseded/`); no cached-mirror field exists. Line-precise on the `status:` key via `_yaml.frontmatter_key_lines` (fn-82.3) — the violation IS the key, so the finding anchors on its file line and survives `--since-staged`. See [`04-surfaces/05-intent.md § 5`](../04-surfaces/05-intent.md#5-acceptance-gates-and-bidirectional-link-verification). Templates and existing draft files were stripped in the same sweep that registered this code.
- `IL014` bundle member unscheduled — a `/abcd:intent plan` bundle whose members are all in the same phase but at least one member is not yet scheduled, so the same-phase guarantee cannot be confirmed at plan time. Severity: info (fixed — see §2; it can never be promoted to a blocker, an unscheduled member simply cannot be confirmed). **Delivered** in fn-8 task .6 (`BundleLinter`).
- `GL001` undefined term (truly unknown candidates — backtick tokens, qualified IDs in `glossary_terms_used` without canonical file, grill-emitted candidates not added). Severity: warn. Tightened scope: tokens listed in `forbidden_synonyms` → GL002; capitalised tokens whose lowercase form matches a glossary entry → GL005. Delivered in fn-3 task .5.
- `GL002` forbidden synonym — intent body uses a value listed in any term's `forbidden_synonyms`. Severity: blocker. Delivered in fn-3 task .5.
- `GL003` cross-context collision — intent body uses a term existing in multiple bounded contexts AND intent lacks `contexts:` frontmatter. Severity: warn. Delivered in fn-3 task .5.
- `GL004` non-stable term in promoted intent — intent in `planned/` references a term with `status: draft` OR `status: deprecated` (scope expanded from draft-only per round-3 Major #4). Severity: blocker. Delivered in fn-3 task .5.
- `GL005` missing glossary citation — intent body uses a recognised glossary term (by display_name or alias, word-boundary match, case-insensitive) but `glossary_terms_used` frontmatter omits the qualified ID. Severity: blocker for promoted intents; warn for drafts; suppressed-as-info for `prd_grandfathered: true` intents (pre-fn-3 grandfather, round-8r5 Critical #1). Distinct extractor from GL001. Delivered in fn-3 task .5.
- `GR001` intent has acceptance but never grilled — intent has populated `## Acceptance` but `grill_session_id` is null. Severity: warn. Delivered in fn-3 task .5.
- `GR002` intent in `planned/` without PRD — `prd_path` is null OR file missing OR `prd_grandfathered: true`. Severity: blocker for newly-promoted; suppressed-as-info for grandfathered. Delivered in fn-3 task .5.
- `GR003` PRD modified post-promotion — PRD with `frozen_at` set has a content hash that no longer matches `frozen_content_hash` (computed per frozen_content_hash recipe in `prd.schema.json`). Severity: blocker. Delivered in fn-3 task .5.
- `GR004` stale planning attempt — PRD with `planning_attempt_id` set but neither a matching spec `## Links` block NOR an in-progress journal for >24h since `frozen_at`. Decision matrix: Links block found → suppress; journal present and <24h old → suppress; journal present and ≥24h old → fire (warn, mention journal age); no journal and >24h since `frozen_at` → fire. Severity: warn. Delivered in fn-3 task .5.
- `GR005` provenance mismatch — PRD `source_intent_hash` does not match current `grilled_intent_hash`, OR PRD `grill_report_hash` does not match on-disk grill-report file. Severity: blocker. Delivered in fn-3 task .5.
- `RV001`–`RV010` — reserved for review lint (front-matter required fields, verdict enum, SHA validation, size cap, etc.) — itd-28 / fn-2 task 3
- `IS001`–`IS010` — reserved for itd-4 issue capture
- `PQ001`–`PQ006` — **delivered** in fn-8 task .8 (`internal/core/lint` / `PromptLinter`). These are the *mechanical* prompt-quality checks fn-8 ships: `PQ001` agent prompt missing `prompt_version` or not valid semver; `PQ002` no matching `### <agent> <version>` heading in `agents/CHANGELOG.md`; `PQ003` `capability_scope` absent or malformed; `PQ004` a `capability_scope.task_classes` token not in `task_classes.json` (closed-enum set-membership); `PQ005` `capability_scope` authored with block-list syntax (must be an inline list); `PQ006` declares `reads_untrusted_input: true` but no canary fixture is *present*. Severity: all blocker. The broader prompt-structural checks of [`05-prompt-quality.md § C`](05-prompt-quality.md) — research-footer / example-I/O structural lint and canary-fixture *execution* (running the canary, not just checking it exists) — are **deferred to a later prompt-test spec** and are not part of the `PQ` codes shipped here.
- `TM001` **RETIRED (tombstoned, never reused).** Was the single collapse code the pre-fn-82.2 `internal/core/lint` terminology adapter folded EVERY `internal/core/lint` schema-validation error into (the tool emitted plain text with no per-error sub-classification, so one code carrying the raw message was the honest registration). fn-82.2 gives `internal/core/lint` a native `--json`/`--codes` interface with per-class TM codes (`TM003`–`TM011` below), so the collapse code is no longer emitted. `TM001` is tombstoned: it is never emitted and never re-allocated to a new meaning.
- **TM allocation table (fn-82.2 R2 — one code per distinguishable `internal/core/lint` error class; recorded BEFORE the native interface landed).** Each distinguishable error class the linter's `_validate_text` produces maps to exactly one NEW code (`TM003`+), OR is classified as a tool HARD-ERROR (exit 2, not a finding). `TM001` retired (above); `TM002` frozen — untouched.

  | # | Error class (`internal/core/lint`) | Native handling | Code / disposition |
  |---|---|---|---|
  | 1 | YAML frontmatter parse failure (`ParseError`) | finding | `TM003` |
  | 2 | Missing required key (`term`/`bounded_context`/`definition`/`aliases`/`forbidden_synonyms`/`status`/`introduced_in`) | finding | `TM004` |
  | 3 | Additional property not allowed | finding | `TM005` |
  | 4 | Bad enum value (`status`) | finding | `TM006` |
  | 5 | Item-level type error (`$.<key>[N]`) + any other generic schema error | finding | `TM007` |
  | 6 | `term` ≠ filename stem | finding | `TM008` |
  | 7 | `bounded_context` ≠ directory name | finding | `TM009` |
  | 8 | `forbidden_synonyms` ∩ `aliases` overlap | finding | `TM010` |
  | 9 | Lifecycle-field asymmetry (`starts_when`/`ends_when` under `status: stable`) | finding | `TM011` |
  | H1 | Schema-load failure (`terminology.schema.json` unreadable/malformed) | HARD ERROR | exit 2, not a finding |
  | H2 | File-read failure (working-tree read `OSError`) | HARD ERROR | exit 2, not a finding |

  Rationale for the finding/hard-error split: classes 1–9 are per-file authoring defects the operator fixes in the file → findings (exit 1). H1/H2 are tool/environment faults that make validation itself impossible (no schema to validate against, or bytes that cannot be read) → hard errors (exit 2), never findings, so the adapter fails closed instead of misreporting a config fault as a clean file. A stdin-mode argument/usage error (missing `--display-path`, non-canonical path, `--stdin` with a positional) is also exit 2 (tool/usage error) and is NOT a lint finding.
- `TM003` terminology term file has a YAML frontmatter parse error — the frontmatter block cannot be parsed (`_yaml.ParseError`); anchored at line 1. Severity: blocker (config-resolvable via `.abcd/config.json` `lint.severity_overrides`/`disabled`, keyed on `TM003`). **Delivered** in fn-82 task .2 (native `internal/core/lint --json`).
- `TM004` terminology term file is missing a required key — a schema-required property (`term`, `bounded_context`, `definition`, `aliases`, `forbidden_synonyms`, `status`, `introduced_in`) is absent. Severity: blocker. **Delivered** in fn-82 task .2.
- `TM005` terminology term file has an additional property not allowed by the schema (`additionalProperties: false`). Severity: blocker. **Delivered** in fn-82 task .2.
- `TM006` terminology term file has a bad enum value — `status` is not one of `draft`/`stable`/`deprecated`. Severity: blocker. **Delivered** in fn-82 task .2.
- `TM007` terminology term file has an item-level or other generic schema error — a list item is the wrong type (`$.<key>[N]`) or any schema error not covered by `TM004`–`TM006`. Severity: blocker. **Delivered** in fn-82 task .2.
- `TM008` terminology `term` ≠ filename stem — the `term` value does not equal the file's stem (canonical term ID). Severity: blocker. **Delivered** in fn-82 task .2.

- `TM002` abcd-owned prose body uses the `core/spec` forbidden synonym `epic` (use `spec`) — a SECOND terminology code (fn-65, itd-43), distinct from `TM001`. Unlike `TM001` (term-file schema validation via the `internal/core/lint` terminology checker), `TM002` is emitted IN-PROCESS by `internal/core/lint`'s `_lint_terminology_prose_files` helper: it scans abcd-owned prose BODY for the forbidden synonym `epic` (ONLY `epic` — not the other `core/spec` `forbidden_synonyms` `sprint`/`milestone`/`project`/`feature`, which are common English words; broad multi-synonym prose enforcement is future work) and flags each unexempt occurrence. Severity: blocker (config-resolvable). The CANONICAL in-scope path set is the single source of truth (`docs/**/*.md`, `commands/abcd/**/*.md`, `.abcd/development/brief/**/*.md`, `.flow/README.md`, root `README.md`, `agents/README.md` + nested `agents/**/README.md`, `intents/README.md`, the live-intent corpus `drafts/`+`disciplines/`+`planned/*.md`, and `skills/abcd-intent-grill/SKILL.md`); historical records (intents `shipped/`+`superseded/`, `decisions/adrs/**`, `research/**`) and terminology term files are EXCLUDED by path. FIVE machine-checkable exemptions: (i) a line containing the external token `flow-next:epic-review` / `epic-review` / `flowctl` passes; (ii) historical records excluded by path; (iii) terminology term files excluded by path (the `forbidden_synonyms` declaration stays owned by `TM001`); (iv) fenced (triple-backtick) and inline (single-backtick) code spans are masked — which is why naming the forbidden noun inside a code span here does NOT fire; (v) YAML frontmatter is not scanned (BODY only — `slug:` / `glossary_terms_used:` machine fields are not prose). Dispatched through BOTH `internal/core/lint` entrypoints — `dispatch` (working-tree bytes) and `dispatch_staged` (`--since-staged`, staged-blob bytes; findings carry a 1-based `line > 0` so they survive the staged-line filter). **Delivered** in fn-65 task .1.
- `XD001` terminology drift, `XD002` schema/state contradiction, `XD003` reservation conflict, `XD004` premise drift, `XD005` scope leakage, `XD006` reference rot, `XD007` acknowledgement gap — reserved by itd-48 (which superseded itd-31); the mechanical-half codes (`XD002`, `XD006`, `XD007`) are **deferred** post-fn-29 (see `.work/issues.md` `[fn-29 follow-up]`)
- `MQ001` per-page quotation budget exceeded (>5% per page OR >150-word contiguous span); `MQ002` cumulative source coverage exceeded (>25% deduplicated coverage across all pages citing one `source_hash`, span-level dedup, configurable threshold). Severity: `MQ001` warn; `MQ002` per-finding at emission — warn in [15%, 25%), blocker ≥25% (thresholds configurable via `.abcd/memory/config.json` `quotation_budget`; a blocker fires only when UNAMBIGUOUS single-source attribution alone reaches the block threshold — ambiguous multi-source attribution caps at warn). ## 2. Severity model

Three severities, configurable per-code:

| Severity | Meaning | Default behavior |
|---|---|---|
| `blocker` | Cannot promote / cannot commit / cannot merge depending on context | Non-zero exit; CI fails; `internal/core/lint --promote-check` blocks `drafts/` → `planned/` |
| `warn` | Surface to user; do not block | Non-zero exit on standalone run; informational on `--promote-check` |
| `info` | Surface only when `--verbose` | Always exit 0 |

**Configuration**: `.abcd/config.json` extends with a `lint` block:

```json
"lint": {
  "severity_overrides": {
    "GL001": "warn",
    "GL002": "blocker"
  },
  "disabled": []
}
```

The defaults are baked into each lint module; user overrides via `severity_overrides`. Disabling a code requires explicit listing in `disabled` (no silent suppressions).

`.abcd/config.json` is optional — it does not exist by default. Every linter (the intent, prompt, and orchestrator surfaces of `internal/core/lint`) tolerates its absence: a missing file (or a missing `lint` block) resolves to an empty config `{}`, so every code keeps its baked-in default severity and nothing is disabled. The block above is the expected shape *when a project chooses to add overrides* — `severity_overrides` is a map of code → severity, `disabled` is a list of codes; both default to empty. `IL011`/`IL014` ignore both (see the fixed-severity exception below).

**Command-specific exit-code exception — `/abcd:memory lint` (fn-39 .3).** The severity table's "non-zero exit on standalone run" for `warn` is the *standalone lint grammar* (direct `internal/core/lint` invocations, `--corpus`, the CI/pre-commit orchestrator). `/abcd:memory lint` is a curator advisory surface and computes its OWN exit code: blocker → nonzero; warn-only → exit 0 (findings reported, NON-blocking); info never affects exit. This divergence is INTENTIONAL — memory-lint warns are curator guidance, not CI gates — and is implemented without `internal/core/lint` (whose standalone grammar returns 2 on warns); it is asserted in `tests/abcd/test_memory_internal/core/lint` so the difference stays explicit, not a bug. Severity *semantics* (what blocker/warn/info mean, overrides, `disabled`) are unchanged — only the exit mapping differs on this one command surface.

**Fixed-severity exception** — `IL011` and `IL014` honour neither `severity_overrides` nor `disabled`. `IL011` (cross-phase bundle) is *always* a hard `blocker`: a bundle that spans phases cannot ship as one spec, and no configuration may demote that. `IL014` (bundle member unscheduled) is *always* `info`: the same-phase guarantee genuinely cannot be confirmed when a member is unscheduled, so the code can never be promoted to a blocker. Both short-circuit severity resolution and the `disabled` check before either consults config — a `severity_overrides` or `disabled` entry naming `IL011` or `IL014` is silently ignored.

## 3. CI integration

Four trigger points:

1. **Pre-commit hook** (opt-in via `.pre-commit-config.yaml`): runs `internal/core/lint --since-staged`. Fails the commit on any `severity: blocker` finding.

   **`--since-staged` mode (fn-18 T2).** The orchestrator computes the staged file set itself via `git diff --cached --name-only --diff-filter=ACMR -- '*.md'`, lints each as a STAGED BLOB read via `git show :<path>` written to a temp file, and filters findings to lines actually staged (parsed from `git diff --cached -U0`). Configured with `pass_filenames: false` + `always_run: true` so the orchestrator owns file-set discovery (rejecting positional arguments to avoid double-linting against working-tree positionals) and the hook still runs on deletion-only commits.

   - **Path-preserving dispatch (option 2).** Each linter is constructed with `intent_path=<display-path>, content_path=<tmp>, repo_root=<real-repo-root>`. The DISPLAY path drives classification / `_detect_location()` / `Finding.file`; the TEMP path supplies bytes; the REAL repo root keeps `_load_glossary(repo_root)` and `.abcd/config.json` lookups working. `IntentLinter`, `SpecLinter`, and `PromptLinter` each accept a `content_path` / `content_paths` kwarg for this dispatch.
   - **PromptLinter `content_paths` contract.** `PromptLinter` accepts a batched `content_paths: dict[Path, Path]` (display→temp). `internal/core/lint` constructs `prompt_files` and `content_paths` from the same source iterable so their keys are byte-equal — no `Path.resolve()` drift, no key-lookup misses.
   - **File-level suppression policy.** Findings with `line == 0` on a real file are SUPPRESSED UNCONDITIONALLY in `--since-staged` mode, even when other lines on the same file are staged. The prior split (keep-if-any-staged-line) is removed; deferred-upgrade candidates are tracked in `.work/issues.md` so each emitter is queued for promotion to line-precise.
   - **Synthetic / no-source findings ALWAYS fire** (their `file` is empty/None — not tied to any staged line).
   - **Path-normalization-failed → HARD ERROR.** Any display path that reaches dispatch as an absolute path triggers `_assert_repo_relative` and raises — never a silent fall-back. The invariant is that staged-mode dispatch operates on REPO-RELATIVE paths (matches `git diff --cached` output) so the staged-line filter can index findings by their `file` string directly.
   - **Terminology subprocess adapter — staged-blob mode (fn-24.3 strategy α; native JSON envelope since fn-82.2).** Under `--since-staged`, the terminology subprocess adapter (`_lint_terminology_files`) pipes the staged blob — `git show :<rel>` bytes — to `internal/core/lint --stdin --display-path <canonical> --json` with `--touched-line <N>` flags for each staged hunk line. Since fn-82.2 the linter emits the canonical `§4` JSON envelope (not plain `OK`/`FAIL` text): the adapter parses the envelope and consumes each finding's own per-class TM code (`TM003`–`TM011`) verbatim — no adapter-only code translation. The linter validates the staged content against the canonical display path (so `term ≠ filename stem` / `bounded_context ≠ directory name` operate on the right name) and emits each finding with a position-derived file-relative `line` per the failure-mode mapping in `internal/core/lint`'s module docstring: missing required key → line of frontmatter start / adjacent present key; bad enum / mismatch keys → line of the offending key; `forbidden_synonyms ∩ aliases` overlap → deterministic selection (touched aliases entry → touched forbidden_synonyms entry → `aliases:` key line as the default). Findings then flow through the standard staged-line filter — a violation on a staged line surfaces; an unchanged pre-existing overlap (off-stage) is dropped by `_classify_and_filter`. The non-`--since-staged` (CI) path keeps the file-path subprocess invocation (`--json`); OS-native paths are accepted and the canonical form is computed for the finding's `file`. **Fail-closed (fn-82.2):** an exit that is neither 0 (clean) nor 1 (findings emitted with a parseable envelope) — an unparseable stdout, a missing/ malformed envelope, or an exit-2 tool/config hard error — is surfaced as a hard adapter error (a synthetic `<aggregate>` finding that always fires under `--since-staged`), NEVER a silent pass and NEVER a fabricated collapse code. The legacy working-tree-read fallback documented in the original fn-18 T2 AC2 is RETIRED by fn-24.3 — the `.work/issues.md` 2026-05-24 terminology entry is the seed for this retirement and is marked resolved on landing.

2. **`/abcd:intent plan <itd-N>` promotion gate**: runs `internal/core/lint --promote-check <intent>` before promoting `drafts/ → planned/`. Blocks on `severity: blocker`. Surfaces `severity: warn` for review but does not block.
3. **CI workflow** (`.github/workflows/lint.yml`): runs `internal/core/lint --since <base>` over **only the lines changed in the PR** — never a blanket repo scan, never a whole-file scan.

   **`--since <base-ref>` range mode (fn-81.7).** The CI tier is LINE-SCOPED, sharing the SAME scope filter (`_classify_and_filter`) and hunk parser (`_parse_hunk_lines`) as the pre-commit `--since-staged` tier. The compared range is `merge-base(base-ref, HEAD)..HEAD`; the orchestrator computes its OWN changed-`.md`-file set (`git diff --diff-filter=ACMR -z <merge-base>..HEAD -- '*.md'`) and per-file changed lines (`git diff -U0 <merge-base>..HEAD`), then reads each file as its `HEAD:<path>` BLOB — NOT the working tree — and reports only findings on lines the PR changed. A changed file with uncommitted working-tree edits is a HARD error (fail loud, never silently lint uncommitted bytes); an unresolvable base ref or diff failure is a HARD error (exit 2), never a fail-open empty scan. `--since` rejects positional arguments and is mutually exclusive with `--since-staged` (each owns file-set discovery from a different source). Fails CI on any `severity: blocker`; `warn`/`info` are surfaced but do not fail the job.

   **Grandfather cutoff.** Running over only the changed LINES is what grandfathers pre-fn-8 documents (tighter than the old changed-files scope). Existing specs (`fn-1`…`fn-7`) predate itd-37 and carry no `## Modification Grammar` section — a blanket (or whole-file) scan would fire `MG`/`VR`/`SD` blockers on every one of them and make the repo permanently red. The changed-lines gate never fails a historical file's untouched lines: a line authored or materially edited after fn-8 ships inherits the discipline gate; a line merely sitting in the tree, untouched, is never retroactively failed. Backfilling discipline sections into pre-fn-8 specs is a non-goal. A commit touching only clean lines of a legacy file passes BOTH the pre-commit and CI tiers identically — the two are no longer scope-divergent (the fn-81.7 parity fix).

4. **Full-corpus promote-backlink workflow** (`.github/workflows/lint-corpus.yml`, added fn-30 / itd-46): runs `abcd lint --corpus --codes-filter PR000 PR001 PR002` on PR triggers. Unlike the changed-files gate (point 3), `--corpus` enumerates its OWN scan roots — every intent file (all five lifecycle dirs), every native-spec-store spec file, and every issue file — and runs each file's per-type check set, adding the `PR` family (`PR001` on intents, `PR000`/`PR002` on issues). It exists because `PR002` (issue-side back-link) can only be verified against the FULL issue corpus, which a per-file changed-files lint never walks. The CI gate is scoped to the `PR` codes via `--codes-filter`: those codes are net-new in fn-30 so nothing is grandfathered against them, whereas gating the pre-existing IL/GL/GR/MG/VR/SD families over the whole tree would fire on hundreds of grandfathered pre-fn-8 / draft intents and make the job permanently red (the same grandfather rationale that scopes point 3 to changed files). A bare `--corpus` (no filter) still RUNS every family for local/manual full audits. Repo-root discovery is `git rev-parse --show-toplevel` (falling back to `$PWD`). Exit code follows the standalone grammar (blocker → 1, warn → 2, clean → 0); it does NOT fail open on a `PR002` blocker. The per-file pre-commit / changed-files paths are UNCHANGED — `--corpus` and a positional path are mutually exclusive, so corpus mode owns the full-tree promote-bridge triggers while per-file mode covers pre-commit and ad-hoc invocations (`PR001` still fires there on a changed intent; `PR002` does not, as it needs the corpus walk).

The pre-commit hook and CI workflow share the same code AND the same scope filter: a single `internal/core/lint` orchestrator that dispatches to the category-specific modules, aggregates output, and applies the ONE `_classify_and_filter` line-scope filter to both tiers. Both run it over a bounded, LINE-SCOPED set — staged files / staged lines for the hook (via `--since-staged`), `merge-base(base, HEAD)..HEAD` changed files / changed lines for CI (via `--since <base>`) — never the whole repo, never a whole file. The two surfaces are CI-PARITY for line-precise findings (both feed the same filter) and HOOK-RELAXED for file-level findings (`line == 0` suppressed under both `--since-staged` and `--since`, since the file-level suppression lives in the shared filter). Both read git BLOBS, never the working tree: the hook reads the staged index blob, CI reads the `HEAD:<path>` blob.

## 4. Output format

Lints emit machine-readable JSON via `--json` and human-readable text by default. Machine format:

```json
{
  "lint_run": {
    "tool": "internal/core/lint",
    "version": "1.0.0",
    "timestamp": "2026-05-07T17:42:00Z"
  },
  "findings": [
    {
      "code": "GL002",
      "severity": "blocker",
      "file": "intents/drafts/itd-N-foo.md",
      "line": 42,
      "message": "Intent body uses forbidden synonym 'session' (canonical: 'voyage')",
      "suggestion": "Replace 'session' with 'voyage' on line 42 (per terminology/core/voyage.md)"
    }
  ],
  "summary": {
    "blocker": 1,
    "warn": 0,
    "info": 0
  }
}
```

Human format mirrors but adds color (in TTY) and elides the `lint_run` envelope.

## 5. Adding a new lint category

When an intent introduces lint codes in a new category:

1. Add the category to the table in §1 with prefix, tool, purpose.
2. Reserve the numeric range (`AA001`–`AA010` typically).
3. Document each code in the intent's "What's In Scope" section AND inline at the relevant tool module.
4. If the new category warrants a new tool (e.g., `internal/core/lint`), the intent's spec ships the tool; otherwise extend `internal/core/lint`.

The `documentation-auditor` agent (per `01-agents.md`) audits this contract on every disembark — any intent that introduces lint codes without registering them here fails the audit.

## 6. Acceptance (the lint contract is itself acceptance-checked)

- **Given** any intent introduces a lint code, **when** the documentation-auditor runs at disembark, **then** the auditor fails if the code is not registered in §1's table.
- **Given** the lint contract is updated, **when** `internal/core/lint`'s intent, prompt, or terminology checker runs, **then** its `--codes` flag emits the contract from §1 (single source of truth). Since fn-82.2 `internal/core/lint` HAS a native `--codes` (and `--json`) interface emitting per-class TM codes (`TM003`–`TM011`); the `internal/core/lint` terminology adapter aggregates the same `TM003`–`TM011` contract rows the tool emits (field-identical by construction — the adapter delegates to the tool's own `--codes`, never re-declares them) into `internal/core/lint --codes`, and `TM002` (the in-process prose code — see §1) is added by `internal/core/lint` itself. The former single-collapse `TM001` code is RETIRED (§1). `internal/core/flowstate` is a different exception shape (fn-41): it is a **standalone emitter** at pre-commit — `internal/core/lint` never dispatches files to it — but it HAS a native `--codes`, and `internal/core/lint` aggregates the same FS contract row into `internal/core/lint --codes` (the aggregate row delegates to the detector's own contract, so the two registries are field-identical by construction; severity resolves through `.abcd/config.json` like every configurable code).
- **Given** a user runs `/abcd:intent plan <itd-N>`, **when** the promotion gate runs `internal/core/lint --promote-check`, **then** the command reads `.abcd/config.json` `lint.severity_overrides` before deciding blocker vs warn.
